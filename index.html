Sunrisr

Create your own morning ritual

Ember

Ruby | Rails

What do you want to tell yourself every morning?

I love browsing through daily affirmation websites. They're a great source of inspirational wisdom. I built Sunrisr because I've always wanted these sites to have an interactive component. Sunrisr achieves this by giving users a daily quiz on their own list of personal affirmation statements.

The front end is built with Ember, and I've made extensive use of Ember's strength in client-side routing and componentized, state-dependent views. The back end is a Rails API with flat, RESTful routes that are well-suited to communicating with Ember's built-in data store. I have also made use of ActiveModel Serializer & Adapter to maintain consistent formatting of request and response data.

The most important feature in the client is a user's ability to perform their morning affirmations. I built the view states and logic for that first, in case the desired experience necessitated changes to my data model. The original data model held up well during this stage of development. Once a user could perform their daily routine with pre-populated affirmations, it was comparatively easy to build the affirmation-management view where users can customize their own list.

Once these two primary branches were complete, I added some guidance for new users, including a button that redirects from morning ritual to affirmation management when the user hasn't written any affirmations yet. Finally, I applied some light styling to make the app feel like a place for inspiration and personal growth.

Bill-Splitter

Split a restaurant tab into fair shares

Ruby | Rails

PostgreSQL

You should be allowed to treat yourself.

Have you ever been out at a restaurant and wanted to order a more expensive dish than your friends, but didn't want the hassle of divvying up the tab fairly? Have you ever wanted to order that top-shelf whiskey, but stuck to beer like everyone else so the server wouldn't have to print separate checks?

I developed Bill-Splitter to solve this problem. It's a fully responsive single-page web application with a dedicated Ruby on Rails API. Server calls from the client are performed with jQuery/AJAX. Since calculating users' shares of a restaurant bill involves highly relational data, it was a natural choice to have my Rails API store users' data in PostgreSQL.

Since no one brings their laptop to a restaurant, I focused on responsive design from the very beginning. All local testing of the user interface was done in the Chrome inspector's mobile toolbar, and the client app is optimized for viewing on most major mobile devices.

Bill-Splitter is a work in progress. The current version has the essential structure necessary to gather basic information from the user, split a restaurant tab into equal shares, and render those shares in the UI. In the next release, I'll deploy an updated version that provides users with share-adjustment options corresponding to the common restaurant payment stories that originally inspired the app.

My stretch goal for the upcoming release is to incorporate the React library into the front end. The Bill-Splitter interface lends itself well to componentization and state-dependent views, and I've been excited to try out React for some time now.

Desert Island

What five photos would you bring?

Node | Express

MongoDB | Mongoose

Handlebars

What five photos would you bring to a desert island?

In today's world of unlimited data plans, gigs upon gigs of storage space, and social media feeds that push new photos off the page faster than you can upload them, we wondered: what if an online photo album only let you upload five images?

Originally developed by myself and three others as a group project at General Assembly, Desert Island is a full-stack web app that lets users upload and share photos. The client app is HTML5/CSS3/JavaScript with Handlebars templates for dynamically rendering content.

The back end is a Node/Express API that uses the aws-sdk module to upload users' photos to a dedicated file bucket on Amazon Web Services' S3 cloud storage system. File metadata is stored in MongoDB via Mongoose middleware and mLab. Before uploading files to AWS, our API validates the file type (photos only), size (10MB max), and number of uploads so far (5 max per user).

Check out the client on GitHub for more of Desert Island's development story, or try the app itself!

Tic-Tac-Toe

My take on a classic game

JavaScript

jQuery | AJAX

Bringing clean code and minimalist design to a classic game.

Building a Tic-Tac-Toe app is something of a rite of passage for developers. I knew that I wanted my version to be visually simple, so I opted for a minimalist layout that ultimately changed very little from my original wireframes. The final product reflects this approach with its all-lowercase typography, laconic directions, and a mild, coffee-and-cream color scheme.

Keeping the app stylistically simple allowed me to focus on what I felt was most important: writing clean front-end JavaScript that conforms to best practices of modern web development. The client application shows excellent separation of concerns, with authentication functions housed separately from game functions. All the code is highly readable, with semantic function names and well-placed comments that make it clear to other developers what the program is doing and how.

Since developing Tic-Tac-Toe, my attention has been drawn away to other projects, but there is still a lot I would like to do with this app. It currently supports two players competing on a single device; I would like to allow for competition on separate devices, as well as an in-game chat feature. A simple game like Tic-Tac-Toe also makes an excellent testing ground for front-end frameworks, so I may come back to this project as a platform for teaching myself Angular or Backbone in the future.
